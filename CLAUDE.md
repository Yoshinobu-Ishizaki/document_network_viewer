## document_network_viewer

A locally-run Python web app for browsing a collection of documents (PDF and Markdown) via an interactive drill-down graph network.

### Core Features

- **Graph view**: Interactive network graph showing documents organized into a 3-level hierarchy. L1↔L1 edges weighted by shared subcategory topics; L2↔L2 edges weighted by document keyword similarity.
- **Drill-down navigation**: Double-click a group node to expand into its children. Breadcrumb navigation goes back up at any level.
- **Document view**: Clicking a document node opens its content in a resizable side panel.
- **Keyword search**: Search bar in the header greps through all document text and navigates to matching documents.
- **Filtering**: Pre-defined attribute filtering ability for categories and documents to be displayed. 
- **Subcategory management**: Right-click an L2 node to rename or merge it into another subcategory.
- **Pre-processing pipeline**: A separate CLI script categorizes documents using a configurable LLM and writes a cached index.

### Architecture

#### Backend: FastAPI (`app.py`)
- Serves the REST API and static frontend files.
- Endpoints:
  - `GET /api/graph` — returns the full graph structure (JSON)
  - `GET /api/doc/{filename}` — returns rendered document content (Markdown as HTML, PDF as `<pre>` text)
  - `GET /api/search?q=<keyword>` — grep-based full-text search; returns `{matches: [{filename, snippet}]}`
  - `PATCH /api/subcategory` — rename an L2 subcategory in index.json
  - `POST /api/subcategory/merge` — merge one L2 subcategory into another
  - `POST /api/quit` — terminate the server process group
- Reads only from `index.json` at runtime. Never reads raw document files.

#### Frontend: Static HTML + JavaScript (`static/`)
- Single HTML page served by FastAPI.
- Uses **vis.js Network** (CDN v9.1.9) for the interactive graph.
- Split-view layout: graph left, document panel right (resizable drag handle).
- Drill-down: root shows L1 nodes → double-click → L2 nodes → double-click → L3 doc nodes → single-click → doc panel.
- Breadcrumb navigation, right-click context menu on L2 nodes.
- Keyword search bar with floating results panel.

#### Preprocessing Script: `preprocess.py`
- `uv run preprocess.py` — incremental mode (default)
- `uv run preprocess.py --rebuild` — full rebuild, resets UI edits

**Incremental mode**: if `index.json` exists, only newly found files are sent to the LLM. Existing doc assignments are read from the current `index.json`, preserving any UI renames/merges.

**Rebuild mode**: regenerates `index.json` from cache. LLM calls still skipped for unchanged files.

- LLM assigns each doc to an L1 category and an L2 subcategory.
- Results cached in `.cache.json` (keyed by file MD5 hash).
- Subcategory constraints enforced after categorization (see config.yaml).
- Extracted text written to `.text_cache/` for keyword search.
- Semantic edges computed: L2↔L2 via keyword Jaccard, L1↔L1 via shared subcategory topic words.
- Semantic distance is calculated for each docs/categories. The algorithm for semantic distance calculation can be chosen from a couple of candidates when `--algo` options is specified.

#### LLM Provider (`LLMClient` in `preprocess.py`)
Configured via `.env` (copy `.env.sample` → `.env`):
```
LLM_PROVIDER=anthropic    # anthropic | openai | gemini
ANTHROPIC_API_KEY=...
ANTHROPIC_MODEL=claude-haiku-4-5-20251001
OPENAI_API_KEY=...
OPENAI_MODEL=gpt-4o-mini
GEMINI_API_KEY=...
GEMINI_MODEL=gemini-1.5-flash
```
Default provider is `anthropic`. OpenAI requires `uv add openai`; Gemini requires `uv add google-generativeai`.

### Configuration: `config.yaml`

```yaml
categories:
  - Jigs & Tools
  - Manufacturing
  - Programming
  - Meeting & Business Trip
  - Other Stuff

subcategory_constraints:
  max_subcategories: 8         # hard upper limit per L1
  min_docs_per_subcategory: 2  # merge smaller L2s into nearest neighbor
```

Remove the `subcategory_constraints` block to disable constraint enforcement.

### Data

- Do NOT read raw files in `data/` at app runtime. The app only reads `index.json`.
- `preprocess.py` is the only component that reads raw document files.
- `data/` folder is read-only. Do not save any additional files into it.
- `data/` is gitignored. 
- `.text_cache/` is auto-generated by preprocess.
- Any files created and used by `preprocess.py` or `app.py` should be saved in `.local` folder except `.env`. So that, `index.json` is actually saved as `.local/index.json`.

### Python

- Use `uv` to run scripts and add packages: `uv run <script>`, `uv add <package>`.
- Python >= 3.13.
- Core dependencies: fastapi, uvicorn, anthropic, markdown, python-frontmatter, pyyaml, pypdf, python-dotenv.

### Project Structure

```
document_network_viewer/
├── app.py              # FastAPI backend
├── preprocess.py       # Preprocessing CLI (LLM categorization + index build)
├── config.yaml         # L1 categories + subcategory constraints
├── .env.sample         # LLM provider configuration template
├── pyproject.toml
├── data/               # gitignored
├── .local/             # gitignored local file folder 
│   └── index.json      # Generated by preprocess.py
│   ├── .cache.json     # LLM result cache (keyed by file hash)
│   ├── ui_state.json   # UI state file for app.py
│   ├── .text_cache/    # Extracted text for keyword search
│   │   └── *.md / *.pdf    # Source documents
└── static/
    ├── index.html
    ├── main.js
    └── style.css
```

### Running the App

```bash
# Step 1: Copy and fill in .env (if using non-Anthropic LLM or custom model)
cp .env.sample .env

# Step 2: Preprocess documents (generates index.json)
uv run preprocess.py          # incremental: only new files
uv run preprocess.py --rebuild # full rebuild from cache

# Step 3: Start the web server
uv run app.py
# App available at http://localhost:8000
```
## Development cycle

When use asks to "tackle todo", "handle todo", or similar meaning, 

- Read @TODO.md and do for each entry written in that one by one. 
- Todo entry is listed by "- [ ]" 
- Ignore entry by "- [x]" because it have been done.
- Handle entry sequentially. The entry nearest to top of the file should be handled first.
- For each entry, always make plan with plan mode.
- When all implementation is done, ask user to approve its result and ask to mark the entry done.
- If user approve, check entry done which means change "- [ ]" to "- [x]".
- Finally ask user to commit changes.
- Then, ask user to move to next entry if there is such one left.
